- **Database**: Drizzle ORM with Neon (Serverless PostgreSQL)
- **Authentication**: Clerk
- **Styling**: Tailwind CSS 4 with PostCSS
- **UI Library**: shadcn/ui + Base UI
- **Package Manager**: npm

⚠️ **CRITICAL**: This is Next.js 16 with breaking changes from 13/14/15. Always refer to the docs before implementing new features.

## 🛠️ Development Workflow

### Before Starting Work

1. Read relevant documentation from `/docs` directory
2. Check the TypeScript compiler for errors: `npm run lint`
3. Ensure database migrations are up to date
4. Review existing patterns in similar files

### Creating New Features

1. **Define Types First**
   - Create TypeScript interfaces for data structures
   - Define component prop types
   - Create API request/response types

2. **Build Database Layer (if needed)**
   - Add schema to `db/schema.ts`
   - Create/run migrations
   - Export inferred types

3. **Create API Routes (if needed)**
   - Place in `app/api/[resource]/route.ts`
   - Follow RESTful conventions
   - Always include authentication checks
   - Handle errors with try/catch

4. **Build Components**
   - Server Components for data fetching
   - Client Components for interactivity
   - Pass data via props, not client-side fetching
   - Type all props with interfaces

5. **Style with Tailwind**
   - Use utility-first approach
   - Support dark mode
   - Ensure responsive design
   - Use semantic color names

6. **Test and Validate**
   - Verify TypeScript has no errors
   - Run ESLint: `npm run lint`
   - Test authentication flows
   - Check responsive design

## ✅ Code Review Checklist

Before submitting code for review, ensure:

- [ ] **TypeScript**: No `any` types, all functions typed
- [ ] **Async**: All async operations handled with try/catch
- [ ] **Errors**: Meaningful error messages and proper HTTP status codes
- [ ] **Auth**: Protected routes verify user identity
- [ ] **Styling**: Responsive design, dark mode support
- [ ] **Accessibility**: ARIA labels, semantic HTML
- [ ] **Comments**: Explain WHY, not WHAT (complex logic only)
- [ ] **Performance**: No unnecessary re-renders, optimized queries
- [ ] **Linting**: `npm run lint` passes without errors
- [ ] **Database**: Migrations created, types exported
- [ ] **Security**: No exposed secrets, SQL injection prevention
- [ ] **Testing**: Forms validate input, error states display

## 🚀 Common Patterns

### Protected Page

```typescript
// app/dashboard/page.tsx
import { auth } from "@clerk/nextjs/server";
import { redirect } from "next/navigation";
import { fetchUserData } from "@/lib/db";

export default async function Dashboard() {
  const { userId } = await auth();
  
  if (!userId) redirect("/login");
  
  const data = await fetchUserData(userId);
  
  return <DashboardContent data={data} />;
}
```

### API Endpoint

```typescript
// app/api/resource/route.ts
import { auth } from "@clerk/nextjs/server";
import type { NextRequest, NextResponse } from "next/server";

export async function GET(request: NextRequest): Promise<NextResponse> {
  try {
    const { userId } = await auth();
    if (!userId) return Response.json({ error: "Unauthorized" }, { status: 401 });
    
    const data = await fetchResource(userId);
    return Response.json({ success: true, data });
  } catch (error) {
    console.error("Error:", error);
    return Response.json({ error: "Internal server error" }, { status: 500 });
  }
}
```

### Reusable Component

```typescript
// components/ResourceCard.tsx
import type { Resource } from "@/db/schema";
import { clsx } from "clsx";

interface ResourceCardProps {
  resource: Resource;
  onDelete: (id: string) => Promise<void>;
  className?: string;
}

export function ResourceCard({ resource, onDelete, className }: ResourceCardProps) {
  const handleDelete = async () => {
    if (confirm("Delete this resource?")) {
      await onDelete(resource.id);
    }
  };

  return (
    <div className={clsx("p-4 bg-white rounded border", className)}>
      <h3 className="font-semibold">{resource.title}</h3>
      <button onClick={handleDelete} className="text-red-600">Delete</button>
    </div>
  );
}
```

### Client Component with State

```typescript
// components/ResourceForm.tsx
"use client";

import { useState } from "react";
import { Button } from "@/components/ui/button";

interface ResourceFormProps {
  onSubmit: (data: unknown) => Promise<void>;
}

export function ResourceForm({ onSubmit }: ResourceFormProps) {
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const handleSubmit = async (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    setError(null);
    setIsLoading(true);

    try {
      const formData = new FormData(e.currentTarget);
      await onSubmit(Object.fromEntries(formData));
    } catch (err) {
      setError(err instanceof Error ? err.message : "An error occurred");
    } finally {
      setIsLoading(false);
    }
  };

  return (
    <form onSubmit={handleSubmit} className="space-y-4">
      <input type="text" name="title" required className="w-full border p-2" />
      {error && <div className="text-red-600">{error}</div>}
      <Button type="submit" disabled={isLoading}>
        {isLoading ? "Saving..." : "Save"}
      </Button>
    </form>
  );
}
```

## 🤔 Common Questions

**Q: Should I fetch data in a Client Component?**
A: No. Fetch in Server Components and pass data as props. Client-side fetching is slower and prevents SSR benefits.

**Q: When do I use `type` vs `interface`?**
A: Use `interface` for object shapes (can extend), `type` for unions/primitives. For component props, prefer `interface`.

**Q: How do I handle authentication?**
A: Use `auth()` from `@clerk/nextjs/server` in Server Components. For Client Components, use `useUser()` hook.

**Q: What's the difference between Server and Client Components?**
A: Server Components fetch data and access secrets; Client Components handle interactivity. Server is default in app/ directory.

**Q: How do I protect an API route?**
A: Always check `auth()` at the start. Return 401 if unauthorized.

**Q: Should I commit node_modules?**
A: No. Always add to `.gitignore` (already configured).
